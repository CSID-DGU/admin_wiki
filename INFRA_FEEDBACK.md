# Infra 문서 검토 피드백

## 검토 범위와 기준

- 검토 대상: `admin_wiki`의 `md/infra/` 및 MkDocs 내비게이션
- 위키 기준 커밋: `origin/main`의 `dbeda34` (2026-07-22)
- 코드 대조 기준: `CSID-DGU/admin_infra`의 `origin/main` `a6433f6` (2026-07-22에 fetch)
- 코드로 확인한 사실과 운영 환경에서 다시 확인해야 할 사실을 구분했다. 비밀번호·Secret 원문은 검토하거나 기록하지 않았다.

| 구분 | 반영 내용 | 근거 |

## 우선순위가 높은 후속 조치

| 우선순위 | 분류 | 대상 | 제안 | 이유 |
| --- | --- | --- | --- | --- |
| P0 | 설정 결함 | `config-server/Chart`와 배포 workflow | `krb5.kdcHost`를 values schema에 명시하고 CI/CD에서 주입한다. `helm template`으로 빈 KDC를 실패시키는 검증도 추가한다. | ConfigMap은 `.Values.krb5.kdcHost`를 참조하지만 기본 `values.yaml`과 workflow의 `helm upgrade` 인자에는 값이 없다. 빈 값이면 krb5 설정이 불완전하게 렌더링된다. |
| P0 | 보안 | API·운영 매뉴얼 | 인증 없는 config-server API의 허용 출발지, NetworkPolicy/방화벽 규칙, 소유자와 정기 점검을 한 표에 명시한다. | 문서는 내부망 전제를 반복하지만, 실제 차단 제어와 검증 방법이 한곳에 없다. |
| P0 | 복구성 | `design/데이터베이스.md` | 검증된 `nodeport_allocations` DDL, 백업 위치·보존 기간, 복구·reconcile 연습 절차를 버전 관리한다. | 문서는 DDL을 `...`로 남기고 자동 백업이 없다고 밝힌다. 장애 시 필요한 가장 중요한 입력이 문서에 없다. |

| P1 | 문서 분리 | `operations/운영-매뉴얼.md` | 일상 점검, 배포·롤백, 장애 runbook, 상태 일치 점검을 별도 문서로 나누고 첫 페이지에는 의사결정 표만 둔다. | 하나의 긴 문서에 정상 운영과 장애 대응이 섞여 있어 긴급 상황에서 필요한 명령을 찾기 어렵다. |
| P1 | 중복 축소 | `design/시스템-아키텍처.md`와 `operations/API-레퍼런스.md` | 승인·삭제 시퀀스와 자원 소유권 설명은 아키텍처 문서를 기준 원본으로 두고, API 문서는 입력·응답·오류·호출자에 집중한다. | 같은 수명주기 설명이 두 문서에 있어 다음 코드 변경 때 함께 갱신하지 않으면 다시 어긋난다. |
| P1 | 이동·축소 | `design/기초-개념.md` | 계정/NFS/Kubernetes 핵심만 Infra에 남기고, 모니터링·네트워크·실행 환경의 일반 설명은 관련 System 문서로 연결하거나 별도 참고 문서로 분리한다. | 입문 자료와 실제 설계가 한 파일에 길게 섞여 있어 인수인계와 장애 대응 모두에 과하다. |

| P1 | API 운영성 | API 레퍼런스 | 각 endpoint에 소유자, 실제 호출자, 인증·노출 범위, 지원 상태(운영/수동/실험)를 표기한다. | 호출처 미확인 API가 많아 삭제·변경 영향 범위를 판단하기 어렵다. |

## 권장 문서 구조

```text
infra/
├── index.md                         # 역할별 읽기 경로와 문서 소유자
├── overview.md                      # 서비스 경계·용어·빠른 시작
├── design/
│   ├── architecture.md              # 상태 소유권·수명주기·연동 경계의 기준 원본
│   ├── data.md                      # 스키마·백업·복구 계약
│   └── concepts.md                  # Infra에 꼭 필요한 개념만
├── operations/
│   ├── routine.md                   # 일상 점검과 상태 일치 점검
│   ├── deployment.md                # CI/CD·롤백·Helm 변경
│   ├── incidents.md                 # 증상별 진단·복구 runbook
│   └── api-reference.md             # HTTP 계약만
└── integrations/
    └── kdc-setup/                   # config-server와 AD·Kerberos·NFS의 연동 경계
```

이 구조로 옮길 때는 기존 URL을 즉시 없애지 말고, 한 릴리스 동안 이전 문서에 새 위치 링크를 남긴 뒤 삭제하는 편이 안전하다.

## 표현·편집 규칙 제안

| 현재 패턴 | 권장 방식 |
| --- | --- |
| 문장 조각과 `이다`체가 섞임 | 설명문은 `이다`체, 명령 단계는 동사형으로 통일한다. |
| `운영 가이드`, `관리자 매뉴얼`처럼 같은 문서를 다른 이름으로 부름 | 파일명과 탐색 메뉴 이름을 기준 용어로 정하고 전 문서에서 재사용한다. |
| `26-07-02`, `2026-07-06`처럼 날짜 표기가 섞임 | `YYYY-MM-DD`로 통일하고, 과거 이력에는 관련 PR·commit 또는 티켓을 함께 링크한다. |
| IP·NodePort·namespace가 본문 여러 곳에 반복됨 | 비밀값을 제외한 운영 인벤토리 표를 한 곳에 두고 다른 문서에서 링크한다. |
| "별도 담당자 관할"만 적고 다음 행동이 없음 | 담당 영역, 점검 가능한 범위, runbook 링크, 에스컬레이션 조건을 함께 적는다. |

## 구현 순서 제안

1. P0 설정·보안·복구성 항목의 실제 운영값과 담당자를 확정한다.
2. 위키 CI에서 nav와 Markdown 빌드를 막는 검사를 추가한다.
3. `운영 매뉴얼`과 `기초 개념`을 새 문서로 분리하되, 기존 URL에는 안내 링크를 유지한다.
4. API·Helm 문서를 코드 변경 PR의 체크리스트에 포함해 코드와 함께 갱신한다.
