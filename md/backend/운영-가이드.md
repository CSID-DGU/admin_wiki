## 3. 배포 구조

배포는 GitHub Actions에서 main 브랜치에 push될 때 자동으로 실행됩니다. Helm을 사용해 K8s 클러스터에 배포됩니다.

- **Helm 릴리즈명**: `admin-prod`
- **Helm 차트 경로**: `/opt/admin_be/helm/admin-prod`
- **네임스페이스**: `default`

배포 흐름은 아래와 같습니다.

1. `application.yml` 생성 (GitHub Secret에서 주입)
2. Gradle 빌드
3. Docker 이미지 빌드 및 Docker Hub push
4. svmanager SSH 접속 후 `helm upgrade --install` 실행

수동으로 재배포가 필요한 경우 svmanager에서 아래 명령어를 실행합니다.

```bash
export KUBECONFIG=/etc/kubernetes/ci-deployer.conf

helm upgrade --install admin-prod /opt/admin_be/helm/admin-prod \
  --namespace default \
  --set image.tag=latest \
  --set forceReload=$(date +%s)
# image.tag=latest로 고정되어 있으면 helm이 변경을 감지하지 못해 Pod를 재시작하지 않음.
# forceReload에 현재 타임스탬프를 넣어 매번 다른 값을 만들어 강제로 Pod 재시작을 유도함.
```

## 4. GitHub Secrets 설정

`application.yml`과 배포에 필요한 민감한 값은 모두 GitHub Secrets에 저장합니다.

GitHub 레포지토리 → Settings → Secrets and variables → Actions에서 관리합니다.

| Secret 이름 | 설명 |
|-------------|------|
| `APPLICATION` | `application.yml` 전체 내용 |
| `DOCKER_USERNAME` | Docker Hub 계정 |
| `DOCKER_PASSWORD` | Docker Hub 비밀번호 |
| `K8S_HOST` | svmanager 호스트 (`210.94.179.18`) |
| `K8S_PORT` | svmanager SSH 포트 (`8080`) |
| `K8S_USERNAME` | svmanager 계정 (`svmanager`) |
| `K8S_PRIVATE_KEY` | svmanager SSH 개인키 |

### APPLICATION Secret 수정

`application.yml`을 변경해야 할 때는 아래 순서로 진행합니다.

1. GitHub Secrets에서 `APPLICATION` 값을 수정합니다.
2. Notion BE 페이지의 `application.yml` 문서도 함께 업데이트합니다.
3. main 브랜치에 push해 배포를 트리거하거나, 수동으로 재배포합니다.

> `application.yml`을 코드 레포지토리에 커밋하지 않습니다. `.gitignore`에 등록되어 있습니다.

## 5. 자주 쓰는 kubectl 명령어

```bash
# Pod 목록 및 상태 확인

helm history admin-prod

# 이전 버전으로 롤백
helm rollback admin-prod
```

## 6. 장애 대응

서비스 이상이 감지됐을 때 아래 순서로 확인합니다.

1. **Pod 상태 확인** — `kubectl get pods`에서 `Running` 여부와 `RESTARTS` 횟수 확인
2. **로그 조회** — `kubectl logs <pod-name> --tail=200`으로 최근 로그 확인
3. **이전 로그 확인** — Pod가 재시작됐다면 `kubectl logs <pod-name> --previous`로 직전 컨테이너 로그 확인
4. **이벤트 확인** — `kubectl describe pod <pod-name>`의 `Events` 섹션에서 원인 파악
5. **재배포** — 이미지나 설정 문제라면 GitHub Actions 재실행 또는 수동 재배포 (3번 참고)
6. **롤백** — 배포 직후 문제가 생겼다면 `helm rollback admin-prod`로 이전 버전 복구

## 8. 화이트리스트 설정

로그인이나 토큰 인증 없이 접근 가능한 API 경로를 설정하는 방법입니다. Swagger UI, 헬스 체크, 이메일 인증 등 인증이 필요 없는 공개 경로를 추가할 때 수정합니다.

**파일 위치**

```
DGU_AI_LAB.admin_be.global.auth.SecurityWhitelist
```

`UNPROTECTED_PATHS` 리스트에 경로 문자열을 추가합니다. 단일 경로는 `/api/auth/login`처럼 정확히 입력하고, 하위 경로 전체를 허용하려면 `/swagger-ui/**`처럼 `/**`를 사용합니다.

```java
public class SecurityWhitelist {

    public static final List<String> UNPROTECTED_PATHS = List.of(
            "/", "/swagger-ui/**", "/swagger/**", "/v3/api-docs/**",
            "/api/auth/token/**", "/api/auth/users/password/**",
            "/api/auth/login", "/api/auth/register", "/api/auth/reissue",
            "/api/auth/email/**",
            "/auth/callback/**",
            "/actuator/health", "/actuator/info",
            "/api/requests/config/**", // 개발 완료 이후 IP 제한 필요
            "/api/internal/**",        // 인프라 내부망 전용 — 운영 시 네트워크 레벨 IP 제한 필요
            "/api/monitoring/**"       // 공개 모니터링 지표 (인증 불필요)
    );
}
```

개인정보를 다루거나 관리자 기능이 포함된 경로는 절대 추가하지 않습니다. 화이트리스트 추가 시 반드시 팀과 공유하고 PR에 이유를 남깁니다. `/api/requests/config/**`, `/api/internal/**`는 현재 운영 환경에서 네트워크 레벨 IP 제한이 필요한 경로입니다.

## 9. 스케줄러 장애 대응

스케줄러는 매일 정해진 시각에 자동으로 실행됩니다. 실패 시 Slack `#farm-admin` / `#lab-admin` 채널에 `🚨` 알림이 오지만, 알림 자체가 누락될 수도 있으므로 로그를 직접 확인해야 하는 경우가 있습니다.

| 스케줄러 | 실행 시각 | 로그 시작 키워드 |
|---------|---------|----------------|
| `RequestSchedulerService` | 매일 08:00 KST | `🗓️ [스케줄러 시작]` |
| `UserSchedulerService` | 매일 09:00 KST | `👤 [스케줄러 시작]` |

### 실행 여부 확인

Pod 로그에서 스케줄러가 실행됐는지 확인합니다.

```bash
# 오늘 RequestSchedulerService 실행 흔적 확인
kubectl logs <pod-name> | grep "스케줄러 시작"

# 실시간 감시 (스케줄러 실행 시각 전후 모니터링)
kubectl logs <pod-name> -f | grep -E "스케줄러|ERROR|WARN"
```

### 실패 로그 패턴별 확인

```bash
# 계정 삭제 실패 (만료 처리)
kubectl logs <pod-name> | grep "계정 삭제 실패"

# Pod / Ubuntu 계정 삭제 실패 (WARN — DB는 DELETED 처리됨)
kubectl logs <pod-name> | grep "deleteExpiredRequest"

# 만료 예고 이메일 발송 실패
kubectl logs <pod-name> | grep "전 알림 실패"

# 사용자 수명주기 처리 실패
kubectl logs <pod-name> | grep "수명주기 처리 중 오류"

# Hard Delete 실패
kubectl logs <pod-name> | grep "Hard Delete 실패"

# 전체 ERROR 레벨 로그
kubectl logs <pod-name> | grep "ERROR"
```

### 실패 유형별 대응

**`계정 삭제 실패 (ID: ...)` — ERROR**

`RequestExpiryService.deleteExpiredRequest()` 내부에서 DB 트랜잭션이 실패한 경우입니다. Slack 알림도 함께 전송됩니다. Request의 `status`가 여전히 `FULFILLED`로 남아 있습니다. 원인을 확인한 뒤 해당 Request를 수동으로 처리해야 합니다.

```sql
-- 만료됐는데 FULFILLED로 남은 Request 조회
SELECT request_id, ubuntu_username, expires_at, status
FROM requests
WHERE status = 'FULFILLED' AND expires_at < NOW();
```

**`Pod 삭제 실패, 계속 진행` / `우분투 계정 삭제 실패` — WARN**

Infra Server 호출만 실패한 경우입니다. DB는 `DELETED`로 정상 업데이트됩니다. Infra 리소스(Pod, Ubuntu 계정)가 실제로 남아 있을 수 있으므로 Infra 서버에서 수동으로 정리가 필요합니다.

```bash
# Infra 서버에서 특정 Pod 존재 여부 확인
kubectl get pods | grep <pod-name>

# Pod 수동 삭제 (필요 시)
kubectl delete pod <pod-name>
```

**`전 알림 실패` — WARN**

만료 예고 이메일 발송만 실패한 경우입니다. 실제 삭제 처리에는 영향이 없습니다. 반복적으로 발생하면 Gmail SMTP 설정이나 네트워크 연결을 확인합니다.

**`수명주기 처리 중 오류` / `Hard Delete 실패` — ERROR**

`UserSchedulerService`에서 특정 사용자 처리가 실패한 경우입니다. 해당 사용자에만 영향을 주며 다른 사용자의 처리는 계속됩니다. 로그에서 이메일 주소를 확인해 해당 사용자 DB 상태를 점검합니다.

```sql
-- 특정 사용자 상태 확인
SELECT user_id, email, is_active, last_login_at, deleted_at
FROM users
WHERE email = '문제가_된_이메일';
```

**`Redis 중복 체크 실패, 발송 진행` — WARN**

Redis 장애 시 SETNX 체크를 건너뛰고 이메일을 발송합니다(fail-open). 동일 사용자에게 만료 예고 이메일이 중복 발송될 수 있습니다. Redis 연결 상태를 확인합니다.

```bash
kubectl exec -it <pod-name> -- /bin/bash
# Pod 내부에서 Redis 연결 확인
redis-cli -h <redis-host> ping
```

## 10. 메시지 템플릿 관리

이메일·Slack 알림 메시지를 재배포 없이 수정할 수 있습니다. ADMIN 권한 토큰이 필요합니다.

### 전체 목록 조회

```
GET /api/admin/messages
Authorization: Bearer {ADMIN_TOKEN}
```

응답에서 `overridden: true`인 항목은 현재 DB 값이 적용 중입니다. `overridden: false`이면 `messages.properties` 기본값이 사용됩니다.

```json
[
  {
    "key": "email.container.created.subject",
    "defaultValue": "[DGU AILab] 서버 배정 안내 ({0})",
    "currentValue": "[DGU AILab] 서버 배정 안내 ({0})",
    "overridden": false
  }
]
```

### 메시지 수정

```
PATCH /api/admin/messages/{key}
Authorization: Bearer {ADMIN_TOKEN}
Content-Type: application/json

{ "value": "수정할 메시지 내용" }
```

DB에 row가 생성되어 즉시 적용됩니다. 재배포 없이 반영됩니다.

### 기본값으로 초기화

```
DELETE /api/admin/messages/{key}
Authorization: Bearer {ADMIN_TOKEN}
```

DB row가 삭제되고 `messages.properties` 기본값으로 자동 복귀합니다.

## 11. Redis 상태 확인

Redis에 장애가 생기거나 큐가 쌓이는 경우 아래 명령어로 상태를 확인합니다.

### Redis Pod 접속

```bash
# Redis Pod 이름 확인
kubectl get pods | grep redis

# Redis CLI 접속
kubectl exec -it <redis-pod-name> -- redis-cli
```

### 자주 쓰는 확인 명령어

```bash
# Redis 전체 메모리·연결 상태 요약
INFO memory
INFO clients

# Slack 큐 길이 확인 (0이면 정상, 쌓여 있으면 Worker 문제)
LLEN slack:notification:queue
LLEN slack:infra:notification:queue

# Slack 큐 내용 확인 (삭제하지 않고 조회)
LRANGE slack:notification:queue 0 9

# 로그인된 사용자 수 (RT 키 개수)
SCAN 0 MATCH "RT:*" COUNT 100

# 특정 사용자 로그인 여부 확인
EXISTS RT:{userId}

# 만료 예고 이메일 발송 여부 확인
KEYS email:preexpiry:*

# 특정 키 TTL 확인 (초, -1이면 TTL 없음, -2이면 키 없음)
TTL RT:42
TTL email:preexpiry:42:7일:2026-07-15
```

> `KEYS *`는 Redis를 블로킹합니다. 운영 환경에서는 반드시 `SCAN`을 사용하세요.

### Slack 큐가 계속 쌓이는 경우

`LLEN slack:notification:queue` 값이 줄어들지 않으면 `SlackNotificationWorker`가 정상 동작하지 않는 것입니다.

```bash
# Worker 로그 확인
kubectl logs <pod-name> | grep "SlackNotificationWorker\|Slack 메시지"

# 큐가 완전히 막혔을 때 수동으로 비우기 (메시지 폐기)
DEL slack:notification:queue
```

전체 Redis 키 목록과 TTL 정책은 [Redis 키 카탈로그](Redis-키-카탈로그.md)를 참고합니다.
