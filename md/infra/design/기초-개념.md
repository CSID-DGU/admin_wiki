# 기초 개념

이 페이지는 [시스템 아키텍처](시스템-아키텍처.md)를 읽는 데 필요한 배경 지식을 모아 둔 레퍼런스입니다. 아키텍처 문서 본문에서는 용어를 짧게만 풀고, 깊은 설명은 이 페이지로 링크합니다. 처음부터 끝까지 읽어도 되고, 본문에서 링크를 타고 들어와 해당 절만 읽어도 됩니다.

각 절은 같은 형식을 따릅니다. **개념이 무엇인지 → 이 시스템에서 어디에 쓰이는지 → 조심할 함정**의 순서입니다.

## 1. 리눅스 계정과 파일 소유권

### UID / GID

리눅스는 사용자를 이름이 아니라 숫자(UID, User ID)로 식별합니다. 파일 소유권도 마찬가지입니다 — 디스크에 기록되는 것은 "이 파일의 주인은 dongmin"이 아니라 "이 파일의 주인은 UID 2041"입니다. 이름과 숫자의 대응표가 passwd 파일이고, 그룹(GID) 쪽 대응표가 group 파일입니다.

이 사실이 이 시스템 설계의 뿌리입니다. NFS로 공유되는 홈 디렉터리의 소유권은 **UID 숫자**로 기록되므로, 같은 NAS를 쓰는 모든 머신(FARM 물리 서버, K8s Pod)이 **같은 UID 대응표**를 봐야 합니다. 두 머신이 같은 UID를 서로 다른 사람에게 주면, 한 사람이 남의 홈을 자기 것처럼 읽고 쓰는 보안 사고가 됩니다. 그래서 config-server는 UID 배정을 한 곳(`/kube_share`의 passwd 파일)에서만 하고, 이 파일을 모든 Pod가 공유합니다.

### passwd · group · shadow · sudoers

| 파일 | 내용 | 비고 |
|------|------|------|
| `passwd` | 사용자 이름 ↔ UID/GID ↔ 홈 경로 ↔ 셸 | 누구나 읽을 수 있음 (암호는 없음) |
| `group` | 그룹 이름 ↔ GID ↔ 소속 멤버 목록 | |
| `shadow` | 사용자별 암호 해시 | root만 읽을 수 있음. 이 시스템은 SHA-512 해시를 기록 |
| `sudoers` | 누가 sudo(관리자 권한 실행)를 쓸 수 있는지 | 문법 오류 시 sudo 전체가 망가지므로 편집에 주의 |

일반적인 리눅스 머신에서는 이 파일들이 `/etc/` 아래에 있고 `useradd` 같은 명령이 관리합니다. 이 시스템에서는 config-server가 NFS 공유(`/kube_share`)에 있는 이 파일들을 **직접 편집**해 UID 배정 장부로 씁니다. 사용자 컨테이너에는 이 파일이 마운트되지 않고, Pod 생성 시 장부에서 읽은 UID/GID가 환경변수로 주입되어 컨테이너 entrypoint가 그 값으로 컨테이너 안에 사용자를 만듭니다([시스템 아키텍처](시스템-아키텍처.md) 6절).

### chmod 700과 권한 세 자리

`chmod 700`의 세 숫자는 순서대로 소유자/그룹/기타의 권한이고, 각 숫자는 읽기(4)+쓰기(2)+실행(1)의 합입니다. `700` = 소유자만 읽기·쓰기·진입 가능, 그룹과 남은 아무것도 못 합니다. 이 시스템에서 사용자 홈은 전부 `chmod 700`으로 만들어지며, 이것이 (PVC 같은 쿠버네티스 격리 장치 없이) **사용자 간 홈 격리를 실제로 담당하는 유일한 장치**입니다. 자세한 맥락은 [시스템 아키텍처](시스템-아키텍처.md) 6절을 참고합니다.

## 2. NFS와 NAS

### NFS란

NFS(Network File System)는 네트워크 너머 서버의 디스크를 내 로컬 디렉터리처럼 마운트해서 쓰는 프로토콜입니다. 서버 쪽에서 "이 디렉터리를 내보낸다(export)"라고 선언하면, 클라이언트가 그 경로를 자기 파일시스템 트리의 한 지점에 붙입니다(마운트). 붙인 뒤에는 `ls`, `cat`, 파일 저장이 전부 네트워크 왕복으로 처리되지만 사용자 눈에는 로컬 디스크와 같아 보입니다.

NAS(Network Attached Storage)는 이 NFS 서버 역할을 하는 저장 장치(또는 장비)를 가리키는 말입니다. 이 시스템의 NAS는 사용자 홈 디렉터리 전체와 `/kube_share` 계정 파일, 그리고 infra-mysql의 데이터까지 담고 있는 **단일 장애점**입니다.

### root_squash

NFS의 기본 안전장치입니다. 클라이언트 머신의 root(UID 0)가 NFS 마운트 안에서 파일을 만지려 하면, 서버가 그 요청을 권한 없는 익명 사용자(보통 nobody)로 **강등**시킵니다. 클라이언트 한 대가 뚫려 root를 뺏겨도 NAS의 모든 파일을 마음대로 못 하게 막는 장치입니다.

이 시스템에서의 부작용이 하나 있습니다. Pod 안의 프로세스가 root여도 NFS 위에서는 힘을 못 쓰므로, **Pod가 스스로 자기 홈 디렉터리를 만들 수 없습니다**. 그래서 config-server가 Pod를 만들기 전에 NAS에 SSH로 접속해 홈을 미리 만들어 둡니다(mkdir + chown + chmod 700). 이 사전 생성 단계가 빠지면 사용자는 홈 없는 컨테이너를 받게 됩니다.

### NFS 위의 파일 락은 불안정하다

여러 프로세스가 같은 파일을 동시에 고치지 못하게 잠그는 장치가 파일 락(flock 등)입니다. 문제는 NFS 위에서의 락이 별도 데몬(lockd)에 의존하고, 이 경로가 환경에 따라 조용히 오동작한다는 점입니다. "락을 잡았다고 리턴했는데 실제로는 안 잠긴" 상황이 생길 수 있습니다.

그래서 config-server의 `LockedFile`은 NFS 경로의 파일을 편집할 때 **락 자체는 로컬 디스크(`/tmp/cssh_lock*`)에 잡습니다**. config-server가 단일 진입점(같은 머신의 프로세스들만 이 파일을 고침)이라는 전제 위에서, 신뢰할 수 없는 NFS 락 대신 신뢰할 수 있는 로컬 락으로 동시성을 지키는 구조입니다.

## 3. 쿠버네티스 (Kubernetes)

### 클러스터, 노드, 그리고 선언적 모델

쿠버네티스는 여러 대의 서버(노드)를 하나의 컨테이너 실행 풀(클러스터)로 묶는 오케스트레이터입니다. 핵심 사용법은 **선언**입니다 — "이런 스펙의 컨테이너가 떠 있어야 한다"라는 목표 상태를 API로 제출하면, 쿠버네티스가 현재 상태를 목표에 맞추는 일을 대신합니다. 명령을 서버마다 SSH로 실행하고 결과를 파싱하는 방식과의 차이가, 이 시스템이 Docker 스크립트 방식을 버리고 쿠버네티스로 옮긴 이유의 핵심입니다([시스템 아키텍처](시스템-아키텍처.md) 1절).

### Pod

쿠버네티스가 컨테이너를 실행하는 최소 단위입니다. 컨테이너 이미지, 마운트할 볼륨, 환경 변수, 붙일 GPU 개수 같은 것을 하나의 스펙(YAML/JSON)에 적어 제출하면 스케줄러가 노드를 골라 띄웁니다. 이 시스템에서는 **사용자 한 명 = Pod 하나**이고, config-server가 kubernetes-client 라이브러리로 스펙을 만들어 제출합니다. 노드 선택은 쿠버네티스 기본 스케줄러에 맡기지 않고 config-server가 Prometheus 점수로 직접 정해 스펙에 박아 넣습니다.

### Service와 NodePort

Pod의 IP는 재생성될 때마다 바뀌므로, 고정된 접속 지점을 주는 별도 리소스가 Service입니다. 그중 **NodePort 타입**은 클러스터의 모든 노드에 같은 포트(기본 범위 30000~32767)를 열고, 그 포트로 들어온 트래픽을 대상 Pod로 전달합니다. 클러스터 **밖**(연구실 밖 사용자)에서 접속해야 하는 이 시스템의 요구에 맞는 타입입니다.

이 시스템에서는 사용자 Pod의 포트 하나당 NodePort Service 하나가 만들어집니다(ssh 22, jupyter 8888, 신청한 추가 포트). 어떤 NodePort 번호를 누가 쓰는지는 쿠버네티스가 아니라 infra-mysql의 `nodeport_allocations` 테이블이 장부로 관리합니다. 함정은 **배정 범위와 방화벽 개방 범위가 다르다**는 것입니다([시스템 아키텍처](시스템-아키텍처.md) 5절).

### namespace

클러스터 안의 리소스를 팀·용도별로 나누는 논리적 칸막이입니다. 이 시스템은 `ailab-frontend`(화면), `default`(admin_be), `ailab-infra`(config-server와 사용자 Pod), `monitoring`(Prometheus·Grafana)으로 나뉘어 있습니다. kubectl로 조회할 때 `-n <namespace>`를 붙여야 해당 칸의 리소스가 보입니다. 구성 의도는 [운영 가이드](../operations/운영-매뉴얼.md) 네임스페이스 절을 참고합니다.

### 볼륨과 subPath, hostPath

Pod에 저장소를 붙이는 것이 볼륨 마운트입니다. 자주 나오는 두 옵션이 있습니다.

- `hostPath` — 노드 호스트의 특정 경로를 그대로 컨테이너에 붙입니다. 이 시스템의 사용자 Pod는 홈(`/home` — 노드 호스트가 마운트해 둔 NFS)과 GPU 디바이스(`/dev/nvidia*`)를 hostPath로 받습니다.
- `subPath` — 볼륨 전체가 아니라 **볼륨 안의 특정 파일·디렉터리 하나만 골라** 컨테이너의 특정 경로에 붙이는 옵션입니다. 이 시스템의 과거 설계는 `/kube_share`의 passwd·shadow를 subPath로 컨테이너의 `/etc/passwd` 자리에 붙였지만, **현행 Pod 스펙에는 이 마운트가 없고** 환경변수 주입 방식으로 바뀌었습니다([시스템 아키텍처](시스템-아키텍처.md) 6절). 코드 주석에 흔적이 남아 있어 읽을 때 혼동하지 않도록 적어 둡니다.

### PV / PVC

PV(PersistentVolume)는 관리자가 준비한 저장소 조각, PVC(PersistentVolumeClaim)는 "저장소 주세요"라는 요청 객체입니다. Pod는 PVC를 통해 PV를 받아 씁니다. 이 시스템의 초기 설계는 사용자마다 PVC를 만들었지만, 모든 PVC가 결국 같은 NFS 서버를 바라봐서 격리 효과가 없다는 이유로 폐기됐고, 지금은 NFS를 직접 마운트합니다. image-store(사용자 이미지 tar 보관)와 infra-mysql 데이터만 PVC로 남아 있습니다. 폐기 경위는 [시스템 아키텍처](시스템-아키텍처.md) 6절을 참고합니다.

### imagePullPolicy와 Ready

- **imagePullPolicy** — Pod를 띄울 때 컨테이너 이미지를 원격 레지스트리에서 새로 받을지, 노드에 있는 것을 쓸지 정하는 정책입니다. `Always`면 매번 받아오고(느리지만 최신 보장), `IfNotPresent`면 노드에 없을 때만 받습니다.
- **Ready** — Pod가 "떴다"에도 단계가 있습니다. 컨테이너 프로세스가 시작된 것과, 트래픽을 받을 준비가 된 것(Ready)은 다릅니다. config-server는 Pod 생성 후 Ready가 될 때까지 폴링(주기적으로 상태를 물어보기)으로 기다린 뒤에야 NodePort Service를 만들고 성공을 응답합니다. NFS 마운트가 실패하면(FailedMount) Pod는 영영 Ready가 되지 않고, 이 대기가 타임아웃으로 끝납니다.

### RBAC와 ServiceAccount

RBAC(Role-Based Access Control)은 "누가 어떤 리소스에 무엇을 할 수 있나"를 정하는 쿠버네티스 권한 체계이고, ServiceAccount는 사람 아닌 **프로그램**에게 주는 신원입니다. config-server는 자기 ServiceAccount로 쿠버네티스 API를 호출해 Pod·Service를 만들고 지웁니다. 이 권한 부여는 Helm 차트가 배포 시 함께 만듭니다([Helm 차트 레퍼런스](../operations/Helm-차트-레퍼런스.md)).

### Helm

쿠버네티스 배포에 필요한 여러 YAML(Deployment, Service, RBAC…)을 템플릿으로 묶어 값만 갈아끼워 배포하는 패키지 도구입니다. config-server는 `Chart/` 디렉터리의 Helm 차트로 배포되며 `make deploy`가 그 shortcut입니다.

## 4. 모니터링 — Prometheus, exporter, DCGM

### Prometheus와 pull 모델

Prometheus는 서버 지표(메트릭)를 수집·저장하는 모니터링 시스템입니다. 특징은 pull 모델입니다 — 각 서버가 지표를 보내오는 게 아니라, Prometheus가 주기적으로 각 대상의 HTTP 엔드포인트를 **긁어갑니다(scrape)**. 지표를 HTTP로 내어주는 프로그램을 exporter라고 부릅니다.

### DCGM exporter

DCGM(NVIDIA Data Center GPU Manager)은 NVIDIA GPU의 사용률·메모리·온도를 읽는 도구이고, DCGM exporter는 그 값을 Prometheus 형식으로 내어주는 exporter입니다. 이 시스템의 GPU 노드마다 DCGM exporter가 떠 있고, config-server의 노드 선택 점수가 여기서 나온 세 메트릭(`DCGM_FI_DEV_GPU_UTIL`, `DCGM_FI_DEV_FB_USED`, `DCGM_FI_DEV_GPU_TEMP`)으로 계산됩니다.

함정이 하나 있습니다. **exporter가 죽으면 그 노드의 지표가 사라지는데**, config-server의 쿼리는 지표 부재를 0으로 간주하므로(`or vector(0)`), 죽은 노드가 "가장 한가한 노드"로 오인되어 선택될 수 있습니다. 스코어링 상세는 [시스템 아키텍처](시스템-아키텍처.md) 2절을 참고합니다.

### PromQL 최소 문법

Prometheus에 질의하는 언어입니다. 이 시스템을 운영하며 실제로 쓰는 형태는 이 정도입니다.

```promql
# farm5 노드의 GPU 사용률 평균
avg(DCGM_FI_DEV_GPU_UTIL{Hostname="farm5"})
```

- `메트릭이름{레이블="값"}` — 레이블로 대상을 거릅니다. 노드 구분은 `Hostname` 레이블입니다.
- `avg(...)` — 노드에 GPU가 여러 장이면 장별 값이 여러 개 나오므로 평균을 취합니다.
- `... or vector(0)` — 왼쪽 결과가 비어 있으면 0을 반환합니다. "지표 없음"과 "0"을 구분하지 못하게 되는 양날의 검입니다.

### Grafana

Prometheus에 쌓인 지표를 대시보드로 보여주는 시각화 도구입니다. 관리자가 노드 부하를 눈으로 확인할 때 씁니다(`http://210.94.179.18:30080`). PromQL을 직접 날리고 싶으면 port-forward로 Prometheus에 붙습니다([시스템 아키텍처](시스템-아키텍처.md) 2절의 명령 참조).

## 5. 네트워크와 방화벽

### 공인 IP, 사설 IP, NAT

- **사설 IP**(예: `192.168.x.x`, `10.x.x.x`)는 내부망 안에서만 통하는 주소이고, **공인 IP**는 인터넷 어디서나 닿는 주소입니다.
- 내부망의 여러 장비가 몇 개의 공인 IP를 나눠 쓰게 해 주는 변환 장치가 NAT(Network Address Translation)입니다. "공인 IP의 9300번 포트로 들어온 것은 내부 farm6의 30000번으로 보내라" 같은 규칙이 **포트포워딩**입니다.

### pfSense

이 시스템의 경계 방화벽·라우터 소프트웨어입니다. 외부에서 클러스터로 들어오는 모든 트래픽이 여기를 거치며, 두 개의 공인 IP에 대해 서로 다른 정책을 겁니다.

| 공인 IP | 정책 | 뜻 |
|---------|------|-----|
| LAB `210.94.179.18` | 화이트리스트 8개 포트 | 목록에 있는 포트만 통과. 그 외는 차단 |
| FARM `210.94.179.19` | farm6 경유 오프셋 매핑 | 외부 포트 `9300+(NodePort−30000)`을 NodePort `30000~30097`로 변환해 전달 |

오프셋 매핑을 예로 풀면, 사용자 Pod의 ssh가 NodePort 30042에 배정됐다면 외부에서는 `210.94.179.19:9342`로 접속합니다(9300 + 42). 반대로 NodePort가 30098 이상으로 배정되면 대응하는 외부 포트가 **없으므로**, 클러스터 내부에서는 멀쩡히 접속되는데 외부에서만 안 되는 상태가 됩니다. 이것이 [시스템 아키텍처](시스템-아키텍처.md) 5절의 "방화벽 함정"입니다.

## 6. 저장과 동시성

### 트랜잭션과 SELECT ... FOR UPDATE

트랜잭션은 여러 SQL 작업을 "전부 되거나 전부 안 되거나"로 묶는 단위입니다. `SELECT ... FOR UPDATE`는 트랜잭션 안에서 읽은 행에 **잠금을 걸어**, 내가 커밋하기 전까지 다른 트랜잭션이 같은 행을 고치지 못하게 합니다.

이 시스템에서는 NodePort 배정이 이 패턴을 씁니다. 두 개의 create-pod 요청이 동시에 들어와도, 사용 중 포트 목록을 `FOR UPDATE`로 잡고 빈 포트를 골라 insert하기 때문에 같은 포트가 두 사용자에게 배정되는 사고가 나지 않습니다.

### Redis

메모리 기반 키-값 저장소입니다. 이 시스템에서는 사용자 이미지 저장/로드 작업의 진행 상태를 `img:<username>` 키로 기록하는 좁은 용도로만 씁니다. 오래 걸리는 이미지 작업(commit/save)이 지금 어느 단계인지를 API가 조회할 수 있게 하는 상태판입니다.

### SSOT (Single Source of Truth)

같은 정보가 두 군데 기록되면 반드시 어긋나는 순간이 온다는 전제에서, "이 정보의 기준 원본은 여기 하나"라고 정해 두는 원칙입니다. 이 시스템의 대표 사례 두 가지입니다.

- 계정(UID/GID)의 SSOT는 `/kube_share`의 passwd 파일입니다. admin_be가 요청에 uid 값을 실어 보내도 config-server는 무시하고 passwd 기준으로 동작합니다.
- 요청 상태(PENDING/FULFILLED…)의 SSOT는 admin_be DB의 `requests` 테이블입니다. config-server는 상태를 아예 모릅니다.

### soft delete

행을 실제로 DELETE 하지 않고 "삭제됨" 표시만 남기는 방식입니다. admin_be의 요청 상태 DELETED가 이 방식이라, 삭제 이력이 DB에 남아 나중에 추적할 수 있습니다.

## 7. 실행 환경 잡학

### Flask와 gunicorn

Flask는 파이썬 웹 프레임워크로, config-server가 이것으로 짜여 있습니다. Flask에 내장된 개발 서버는 운영용이 아니므로, 운영 배포에서는 gunicorn(파이썬 웹 앱을 여러 워커 프로세스로 돌려 주는 서버)이 Flask 앱을 감싸 실행합니다.

### WAS와 Spring Boot

WAS(Web Application Server)는 웹 요청을 받아 비즈니스 로직을 실행하는 서버의 통칭입니다. 이 시스템의 WAS가 admin_be이고, Java 진영의 프레임워크인 Spring Boot로 짜여 있습니다. 운영 대화에서 "WAS"라고 하면 admin_be를 가리킵니다.

### Swagger

API 명세를 웹 페이지로 보여 주고 그 자리에서 호출해 볼 수 있게 하는 문서 도구입니다. config-server의 Swagger는 `http://210.94.179.18:30082/apidocs/`에 떠 있습니다. 단, [시스템 아키텍처](시스템-아키텍처.md) 1절의 운영 경고대로 이 API에는 인증이 없으므로, Swagger에서의 "Try it out"은 실제 인프라를 조작하는 실탄이라는 점을 기억해야 합니다.
