# 배포본과 레거시 경계

admin_infra 레포에는 현행 운영 코드와 레거시(과거에 쓰였지만 지금은 운영에서 빠진 코드)가 한 디렉터리 안에 섞여 있습니다. 이 페이지는 "무엇이 지금 클러스터에 떠 있고, 무엇이 잔재인가"의 경계를 정의합니다. 레거시의 공통 원칙은 **읽되, 손대지 않는다**입니다. 과거 구조를 이해하는 자료로는 유용하지만, 수정·재배포·강제 정리는 혼란이나 데이터 손상만 만듭니다.

---

## 1. 컴포넌트별 상태 표

| 컴포넌트 | 요약 |
|----------|------|
| `config-server/` | 운영 중인 유일한 서비스 코드입니다. ns `ailab-infra`, NodePort 30082 |
| `pvc-image-chart/`  | image-store PVC. 사용자 이미지 tar 보관용으로 유일하게 남은 PVC입니다 |
| `infra-sql/`  | infra-mysql StatefulSet. `pod_port_db`(NodePort 배정 장부)를 호스팅합니다 |

---

## 2. 컴포넌트별 설명

### config-server — 현행

인프라 조작의 유일한 창구입니다. admin_be의 지시를 받아 Pod(쿠버네티스가 컨테이너를 실행하는 최소 단위)와 NodePort Service(클러스터 밖에서 접속할 수 있게 노드에 고정 포트를 여는 서비스)를 만들고, NFS 계정 파일과 NAS 홈, Kerberos keytab(커버로스 인증에 쓰는 열쇠 파일)을 관리합니다. 상세 구조는 [시스템 아키텍처](시스템-아키텍처.md)를 참고합니다. 단, Deployment 이름 `containerssh-config-server`와 Service 이름 `containerssh-config-service`의 `containerssh-` prefix는 과거 ContainerSSH(SSH 접속을 컨테이너로 연결해 주던 게이트웨이) 시절의 잔재일 뿐이므로 이름에 속으면 안 됩니다.

### pvc-image-chart — 현행

공용 image-store PVC(`pvc-image-store`)를 배포합니다. PVC(PersistentVolumeClaim)는 쿠버네티스에서 스토리지를 요청하는 객체입니다. 사용자별 PVC가 폐기된 뒤에도 사용자 이미지 tar 보관(`/image-store`) 용도로 유일하게 남은 PVC입니다. migrate와 이미지 저장/로드 흐름이 여기에 의존하므로 삭제하거나 새로 만드는 작업은 신중해야 합니다.

### infra-sql — 현행

`ailab-infra`에 infra-mysql StatefulSet(고정 이름과 저장소를 유지하는 Pod 관리 객체)과 ClusterIP Service를 만듭니다. config-server의 NodePort 배정 장부(`nodeport_allocations`)가 여기 있습니다. 이 테이블은 자동 bootstrap되지 않으므로 DB 재구축 시 별도 DDL이 필요합니다.

### pvc-chart (사용자 PVC) — 폐기 (26-07-02)

초기 설계의 사용자/그룹별 홈 PVC 생성 차트입니다. 26-06-19에 폐기가 결정되고 26-07-02에 운영(config-server API 제거)까지 반영됐습니다. 폐기 근거는 두 가지입니다. 사용자 수만큼 PVC/PV 객체가 늘어 관리 부담과 잔해가 누적됐고, 어차피 모든 PVC가 같은 NFS 공간을 바라봐 간접층이 격리를 제공하지 못했습니다. **읽되 손대지 않습니다.** `ailab-infra`에 Terminating PVC 잔해가 남아 있을 수 있는데, 강제 정리는 데이터 손상 위험이 있습니다. StorageClass manifest들은 과거 클러스터 상태를 해석할 때만 참고합니다.

### auth-server — 미배포 레거시

ContainerSSH 시절의 별도 인증 서버(FastAPI)입니다. 매니페스트만 레포에 잔존하며 현재 클러스터에 배포되어 있지 않고 운영 흐름과 무관합니다. **읽되 손대지 않습니다.** `restart_auth.sh`가 가리키는 ns `containerssh`도 레거시라, 이 디렉터리를 "고쳐서 다시 띄우는" 작업은 존재하지 않는 시스템을 되살리는 일입니다.

### containerssh — 이름만 잔존

ContainerSSH gateway를 배포하던 Helm chart입니다. ContainerSSH 자체가 운영 흐름에서 제거되어, 현재는 config-server의 Deployment·Service 이름에 `containerssh-` prefix로만 흔적이 남았습니다. **읽되 손대지 않습니다.** 과거 SSH 게이트웨이 구조(인증 webhook, config webhook)를 이해하는 사료로만 읽습니다.

### guestImage — 참고용

CUDA 게스트 이미지 Dockerfile 모음입니다. 레포에 있지만 실제 사용자 이미지는 별도 소스(`dguailab/decs` 계열)에서 빌드됩니다. **읽되 손대지 않습니다.** 특히 레거시 entrypoint(컨테이너가 시작할 때 실행되는 첫 스크립트) 중 `exit 0`으로 즉시 종료하는 버전이 섞여 있어, 이 디렉터리 기준으로 이미지를 빌드해 배포하면 Pod가 시작 직후 Completed로 죽는 함정이 있습니다.

---

## 3. 판단이 흔들릴 때

- 이름에 `containerssh`가 들어 있다고 레거시인 것은 아닙니다(현행 config-server의 리소스 이름). 반대로 디렉터리가 레포에 있다고 현행인 것도 아닙니다(auth-server, containerssh).
- 기준은 항상 "지금 클러스터의 `ailab-infra`에 떠 있는가"입니다. 문서와 실물이 어긋나 보이면 kubectl 실측을 우선합니다.
