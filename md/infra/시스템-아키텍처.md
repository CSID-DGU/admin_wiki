# 시스템 아키텍처

이 문서는 admin_infra가 담당하는 인프라 계층의 전체 구조를 설명합니다. 읽는 순서는 위에서 아래로 설계되어 있습니다 — **1절**에서 시스템이 왜 이런 모양인지(계층과 책임 경계)를 잡고, **2절**에서 가장 중요한 흐름인 "승인 → Pod 생성"을 따라가고, **3~6절**에서 그 흐름을 떠받치는 각 조각(코드, API, 포트, 계정·스토리지)을 파고들고, **7절**의 배포 뷰로 "무엇이 어디에 떠 있는가"를 지도로 정리합니다.

본문에 나오는 기술 용어(Pod, NodePort, NFS, root_squash, PromQL 등)의 배경 설명은 [기초 개념](기초-개념.md) 페이지에 모아 두었습니다. 용어가 낯설면 그쪽을 먼저 읽고 돌아오는 것을 권합니다.

## 1. 계층 구조와 책임 경계

### 1.1 세 개의 계층

이 시스템 전체는 "무엇이 존재해야 하는가"를 정하는 층과 "실제로 만들고 지우는" 층, 그리고 "만들어진 것이 실제로 올라가 있는" 층으로 나뉩니다. admin_infra는 이 중 가운데, **인프라를 조작하는 층**을 담당합니다.

| 계층 | 컴포넌트 | 책임 |
|------|----------|------|
| 정책 (Control Plane) | admin_be (Spring Boot) | 신청·승인·만료 스케줄링·알림. "무엇이 존재해야 하는가"라는 목표 상태를 자기 DB에 기록해 관리합니다 |
| 실행 (Data Plane) | **config-server (Flask)** | Pod와 NodePort Service 생성·삭제, NFS 계정 파일 기록, NAS에 SSH로 홈 폴더 미리 만들기, Prometheus 기반 노드 선택 |
| 자원 | Kubernetes + NAS + KDC | 실제 컨테이너·스토리지·인증. Pod 홈은 NAS의 NFS 공유가 마운트되며, FARM에서는 krb5 인증이 마운트의 전제 조건입니다 |

이 구분이 실무에서 갖는 의미는 **장부의 위치**입니다. "이 사용자의 요청이 승인됐는가"는 admin_be DB(`requests` 테이블)가, "이 NodePort를 누가 쓰는가"는 infra-mysql(`nodeport_allocations`)이, "이 UID가 누구 것인가"는 `/kube_share`의 passwd 파일이 각각 기준 장부입니다. 어떤 정보가 이상할 때 어느 장부를 열어야 하는지가 계층 구분에서 바로 나옵니다.

### 1.2 왜 config-server를 별도 서비스로 분리했는가

admin_be가 Kubernetes API를 직접 호출할 수도 있었습니다. 그렇게 하면 서비스 하나가 줄고 역조회 같은 통신 왕복도 없습니다. 그럼에도 분리한 근거는 책임 경계입니다.

- **자격 증명 격리** — 인프라 자격 증명(K8s 토큰, NAS SSH 키, KDC 권한)이 config-server 한 곳에만 존재합니다. admin_be가 침해되거나 오동작해도, 인프라를 직접 조작할 수 있는 열쇠는 admin_be에 없습니다. 열쇠를 가진 표면적이 좁을수록 사고의 반경도 좁습니다.
- **독립 배포** — 인프라 계층을 동결한 채 WAS(admin_be)만 독립 배포할 수 있습니다. 이것은 이론상 장점이 아니라 실제로 벌어진 운영입니다 — config-server 동결 기간에 admin_be만 배포하는 운영을 했습니다. 두 계층의 배포 주기가 다를 때 결합되어 있으면 한쪽의 급한 수정이 다른 쪽의 검증 부담을 끌고 들어옵니다.
- **장애 절단면** — 장애 분석 시 "정책 계층 문제인가 실행 계층 문제인가"를 API 경계에서 자를 수 있습니다. 승인 버튼을 눌렀는데 Pod가 안 떴다면, admin_be가 create-pod를 호출했는지(정책 쪽), 호출은 왔는데 config-server가 실패했는지(실행 쪽)를 HTTP 요청 로그 한 줄로 가를 수 있습니다.

대가는 통신 복잡성입니다. 대표적으로 2절 E2E 흐름의 **accept-info 역조회** — 지시를 받는 쪽인 config-server가 반대로 admin_be를 호출해 사용자 설정을 받아오는 경로 — 가 그것입니다. 호출 방향이 양방향이 되면서, 한쪽이 죽었을 때의 증상도 두 가지(지시가 안 감 / 역조회가 안 됨)로 늘었습니다.

### 1.3 왜 Kubernetes Pod인가

FARM에는 관리자가 물리 서버에서 Docker 스크립트로 컨테이너를 만들던 방식이 먼저 있었습니다. 그 방식으로도 컨테이너는 뜹니다. 그런데도 사용자 컨테이너를 Kubernetes Pod로 옮긴 이유는, 이 시스템의 목표가 "컨테이너를 띄우는 것"이 아니라 "**띄우고 지우는 일을 자동화하는 것**"이기 때문입니다. 자동화 관점에서 두 방식의 차이를 항목별로 보면 이렇습니다.

- **생성·삭제가 API 호출입니다.** Pod는 선언적 객체라서 config-server가 kubernetes-client로 스펙을 제출하면 끝입니다. Docker 스크립트 방식이라면 대상 서버마다 SSH로 들어가 명령을 원격 실행하고 그 결과(성공했나? 이미 있었나? 왜 실패했나?)를 문자열로 파싱해야 하는데, 그 자동화는 훨씬 깨지기 쉽습니다.
- **배치 노드를 자동으로 고를 수 있습니다.** 후보 노드들의 Prometheus 부하 스코어를 계산해 가장 한가한 노드에 Pod를 배치합니다(2.3절 스코어링 상세). 스크립트 방식에서는 "어느 서버에 만들까"를 사람이 정해야 했습니다.
- **GPU 할당이 Pod 스펙에 선언됩니다.** admin_be가 내려준 GPU 개수만큼의 디바이스 마운트를 Pod 스펙에 적어 제출하면 되므로, 노드에 들어가 컨테이너 실행 옵션을 손으로 맞출 필요가 없습니다.
- **접속 통로가 리소스 단위로 관리됩니다.** 사용자별 접속 포트는 NodePort Service라는 독립 리소스로 열리고, 회수할 때도 그 리소스를 지우면 됩니다. 방화벽·포트포워딩 설정을 서버마다 손대는 방식보다 개설·회수가 명확합니다.
- **만료 회수가 결정적입니다.** Pod를 삭제하면 그 안의 프로세스·컨테이너가 함께 정리됩니다. "스크립트가 지웠다고 했는데 프로세스가 남아 있는" 종류의 애매함이 리소스 삭제라는 명확한 연산으로 바뀝니다.

물론 공짜는 아닙니다. Kubernetes 자체의 러닝커브가 있고, Pod가 NAS의 NFS 홈과 krb5 인증에 결합되면서 마운트 실패(FailedMount) 같은 새로운 장애 유형이 생겼습니다. 그럼에도 "승인 한 번으로 생성부터 회수까지"라는 자동화 목표에는 선언적 리소스 모델이 맞는 도구라는 것이 이 구조의 판단입니다.

### 1.4 ⚠️ 운영 경고 — 인증 없는 API

config-server API에는 요청자 인증 계층이 없습니다. NodePort(30082)로 노출된 상태에서 `/create-pod`, `/accounts/*`를 외부에서 직접 부를 수 있으면 그대로 인프라 조작이 됩니다. 계정을 만들고, Pod를 띄우고, 남의 Pod를 지우는 일이 전부 HTTP 요청 한 번입니다. 내부망 제한·NetworkPolicy(Pod 간 통신을 제한하는 쿠버네티스 방화벽 규칙)가 이 구조의 **전제 조건**이며, 방화벽 정책을 손댈 때는 30082가 외부에 열리지 않는지 반드시 확인해야 합니다. Swagger(`/apidocs/`)의 "Try it out"도 같은 이유로 실탄임을 기억합니다.

## 2. 승인 → Pod 생성 End-to-End

관리자가 승인 버튼을 누르는 순간부터 사용자가 접속 정보를 받을 때까지의 전체 체인입니다. 이 체인의 어느 마디에서 끊겼는지 확인하는 것이 장애 대응의 출발점입니다.

### 2.1 전체 시퀀스

```mermaid
sequenceDiagram
    participant BE as admin_be
    participant CS as config-server
    participant NAS as NAS/KDC
    participant K as Kubernetes

    BE->>CS: PUT /accounts/users
    CS->>CS: /kube_share passwd·group·shadow 기록 (빈 UID 골라 배정)
    CS->>NAS: SSH로 홈 생성 (mkdir·chown·chmod 700)
    Note over CS,NAS: Kerberos 인증 준비 (별도 담당자 관할)
    CS-->>BE: uid/gid 반환
    BE->>CS: POST /create-pod {username}
    CS->>BE: GET /api/requests/config/{username} (역조회)
    BE-->>CS: 이미지·GPU 노드·추가 포트
    CS->>CS: Prometheus 스코어로 최소 부하 노드 선택
    CS->>K: Pod 생성 → Ready 폴링 대기
    CS->>K: NodePort Service 생성 (ssh/jupyter/추가 포트)
    CS-->>BE: 실제 할당 포트 응답
    BE->>BE: FULFILLED 저장 + 배정 메일 발송
```

큰 그림은 두 번의 호출입니다. admin_be가 먼저 **계정을 만들라**(`PUT /accounts/users`)고 시키고, 성공하면 **Pod를 만들라**(`POST /create-pod`)고 시킵니다. 둘 다 성공해야 FULFILLED로 확정되고 사용자에게 메일이 갑니다. 아래에서 각 마디를 자세히 봅니다.

### 2.2 단계별 상세

#### ① 계정 생성 — `PUT /accounts/users`

파일 락을 잡고, passwd에서 관리 범위 안 가장 큰 UID+1을 새 UID로 골라 배정해 **passwd → group → shadow** 순으로 기록합니다. sudo 허용 명령이 설정되어 있는 경우에만 추가로 `sudoers.d`에 사용자별 정책 파일을 만듭니다(설정이 비어 있으면 sudoers 단계는 건너뜁니다). 중간에 실패하면 그때까지 쓴 passwd·shadow·group 기록을 되돌립니다.

세부 사항 세 가지가 운영에서 중요합니다.

- **락은 로컬에 잡습니다.** 계정 파일은 NFS 위에 있는데 NFS 파일 락(lockd 의존)은 불안정합니다. 그래서 `LockedFile`은 실제 락을 로컬 `/tmp/cssh_lock*` 파일에 잡는 이중 보호를 씁니다. config-server가 이 파일들의 유일한 편집 주체라는 전제 위의 설계입니다([기초 개념](기초-개념.md) 2절).
- **passwd가 UID의 SSOT입니다.** admin_be가 요청에 uid 값을 실어 보내더라도 config-server는 무시하고, passwd 파일 기준으로 빈 UID를 배정해 **응답으로 되돌려 줍니다**. UID의 기준 원본이 두 개가 되는 것을 막기 위해서입니다(6.1절).
- **shadow에는 SHA-512 해시**가 기록됩니다.

#### ② NAS 홈 미리 만들기

NFS export가 `root_squash`(클라이언트 root를 권한 없는 사용자로 강등시키는 안전장치)라서 **Pod 스스로는 자기 홈을 만들 수 없습니다**. Pod 안에서 root여도 NFS 위에서는 힘이 없기 때문입니다. 그래서 config-server가 NAS(운영 배포 기준 `192.168.2.30:6954` — 코드에서는 `NAS_SSH_HOST`/`NAS_SSH_PORT` 환경변수로 주입)에 SSH로 들어가 홈을 미리 만듭니다 — mkdir로 만들고, chown으로 방금 배정한 UID에게 소유권을 주고, chmod 700으로 본인 외 접근을 막습니다. 이 chmod 700이 사용자 간 홈 격리의 실체입니다(6.2절).

#### ③ Kerberos 인증 준비

FARM NAS 홈 마운트에는 Kerberos(krb5) 인증이 전제됩니다. 이 영역은 별도 담당자 관할이라 이 wiki에서는 다루지 않습니다. 관리자가 알아야 할 것은, 이 체인이 끊기면 신규 Pod가 FailedMount로 Ready에 도달하지 못한다는 사실뿐입니다(6.3절).

#### ④ create-pod와 역조회

`POST /create-pod`는 `{"username": ...}` 수준의 최소 입력만 받습니다. Pod를 만드는 데 필요한 나머지 정보 — 어떤 이미지, GPU 몇 개, 어떤 추가 포트 — 는 config-server가 admin_be에 **역조회**해서 받아옵니다. 호출 경로는 클러스터 내부 DNS의 `admin-prod.default` Service를 향한 `GET /api/requests/config/{username}`입니다.

왜 이렇게 설계했는가 하면, 요청 설정의 기준 장부가 admin_be DB이기 때문입니다. create-pod 호출에 설정을 몽땅 실어 보내는 방식이라면 그 순간의 스냅샷이 진실이 되어 버리는데, 역조회 방식은 "만드는 순간의 최신 승인 설정"을 항상 장부에서 읽습니다. 대가는 의존성입니다 — **이 경로가 끊기면 승인은 되는데 Pod 생성만 실패하는** 특유의 증상이 나타납니다. admin_be가 내려가 있거나 `admin-prod` Service 이름 해석이 안 될 때가 그렇습니다.

#### ⑤ 노드 선택

후보 GPU 노드들의 Prometheus 지표로 부하 스코어를 계산해 최소 부하 노드를 고릅니다. 계산식과 폴백은 2.3절에서 따로 다룹니다.

#### ⑥ Ready 대기 → NodePort 생성

Pod 생성 요청이 접수된 것과 Pod가 실제로 뜬 것은 다릅니다. config-server는 Pod가 Ready(트래픽을 받을 준비 완료)가 될 때까지 폴링으로 기다린 다음에야 NodePort Service를 만들고 성공을 응답합니다. 순서가 이런 이유는, 뜨지도 못한 Pod에 접속 포트부터 열어 주면 "포트는 배정됐는데 접속은 안 되는" 반쪽 성공이 사용자에게 전달되기 때문입니다.

Ready 대기 상한은 현재 main 기준 300초(`POD_READY_MAX_WAIT_SEC`)이지만, **대기 시간과 imagePullPolicy는 배포 브랜치에 따라 다르므로 배포본에서 실측**해야 합니다([배포본과 레거시 경계](배포본과-레거시-경계.md)). 이미지가 커서 pull이 오래 걸리면 이 타임아웃에 걸릴 수 있습니다.

#### ⑦ 실패 시 되돌리기

Pod 생성이 실패하면 config-server는 그때까지 만든 것을 되돌립니다 — 잡아둔 NodePort 배정을 해제하고 생성된 Pod를 삭제합니다. admin_be는 만들어 둔 계정을 지우고 요청을 PENDING으로 되돌립니다. 관리자는 원인(이미지 문제, 노드 문제, NAS 문제 등)을 해소한 뒤 재승인하면 됩니다. 이 되돌리기가 완전하지 못한 알려진 결함이 하나 있는데, 2.4절 상태 전이에서 다룹니다.

### 2.3 노드 선택 스코어링 상세

⑤ 단계의 후보는 admin_be가 역조회 응답에 실어 준 GPU 노드 목록이고, admin_be가 gpu_nodes를 주지 않으면 클러스터의 Ready 상태 워커 노드 전체로 폴백합니다(control-plane 노드 제외). 스코어는 DCGM(NVIDIA GPU 지표를 Prometheus로 내보내는 exporter) 메트릭 세 개로 계산합니다. `utils.py`의 실제 PromQL 기준입니다.

| 메트릭 | 의미 | 반영 방식 |
|--------|------|----------|
| `DCGM_FI_DEV_GPU_UTIL` | GPU 사용률 (%) | 노드 내 GPU 평균값 그대로 |
| `DCGM_FI_DEV_FB_USED` | 프레임버퍼(GPU 메모리) 사용량 (MiB) | 평균값 ÷ 1024 (GiB 단위로 축소) |
| `DCGM_FI_DEV_GPU_TEMP` | GPU 온도 (°C) | 평균값 ÷ 100 (미세 가중) |

계산식은 다음과 같고, **점수가 낮을수록 한가한 노드로 선택**됩니다.

```
score = avg(GPU_UTIL) + avg(FB_USED)/1024 + avg(GPU_TEMP)/100
```

단위를 그대로 두면 사용률(0~100)이 지배 항이 되도록 짜여 있습니다. 메모리는 GiB로 축소해 보조 항(수십 GiB 노드에서 0~수십), 온도는 100으로 나눠 소수점 아래 미세 가중(0.3~0.8 수준)입니다. 결과적으로 "사용률이 비슷하면 메모리가 널널한 쪽, 그것도 비슷하면 시원한 쪽"이라는 우선순위가 계수에 박혀 있는 셈입니다.

폴백(대체 경로) 동작이 두 가지 있고, 방향이 서로 반대라는 점이 중요합니다.

- **메트릭 부재 → 0점 (과소평가)** — 쿼리에 `or vector(0)`이 붙어 있어 해당 노드의 DCGM 메트릭이 없으면 그 항은 0으로 계산됩니다. 즉 **exporter가 죽은 노드는 "한가한 노드"로 오인되어 선택될 수 있습니다.** 특정 노드에만 Pod가 몰린다면 그 노드의 exporter 생사부터 확인합니다.
- **쿼리 실패 → 무한대 점수 (배제)** — Prometheus 통신 자체가 실패하면 그 노드는 무한대 점수를 받아 사실상 배제됩니다. 후보 전체가 실패하면 노드를 못 골라 create-pod가 실패합니다(로그에 `no suitable node selected`).

관리자가 점수 재료를 직접 볼 때는 Grafana(`http://210.94.179.18:30080`)를 쓰거나, port-forward 후 PromQL을 날립니다.

```bash
kubectl -n monitoring port-forward svc/monitoring-kube-prometheus-prometheus 9090:9090
# 브라우저 http://localhost:9090 에서 예시 쿼리:
# avg(DCGM_FI_DEV_GPU_UTIL{Hostname="farm5"})
```

### 2.4 요청 상태 전이 (admin_be 관할)

위 승인·실패 되돌리기·만료 흐름에서 요청(request)이 거치는 상태 전체입니다. 상태를 기록하는 기준 장부는 admin_be DB의 `requests` 테이블이며, **config-server는 상태를 알지 못합니다** — 인프라 작업의 성공/실패만 응답하고, 상태 전이는 전부 admin_be가 수행합니다. "상태가 이상하다"는 문제는 따라서 항상 admin_be 쪽 장부의 문제입니다.

```mermaid
stateDiagram-v2
    [*] --> PENDING : 사용자 신청
    PENDING --> PROCESSING : 관리자 승인 시작
    PROCESSING --> FULFILLED : 계정/Pod/포트 생성 + DB 확정 성공
    PROCESSING --> PENDING : 인프라 단계 실패
    PENDING --> DENIED : 관리자 거절
    FULFILLED --> DELETED : 만료 — 인프라 정리 후 확정
    PENDING --> DELETED : 단순 삭제
    DENIED --> DELETED : 단순 삭제
    DELETED --> [*]
    note right of PROCESSING
        알려진 결함 — 인프라 생성까지 성공한 뒤
        마지막 DB 확정 단계에서 실패하면
        인프라는 만든 것 되돌리기를 시도하지만
        상태는 PROCESSING에 방치됩니다.
        남은 거(계정·Pod) 확인 후 수동 정리가 필요합니다.
    end note
```

인프라 실패(계정 생성·Pod 생성 실패)는 만든 것을 되돌리는 정리가 끝난 뒤 PENDING으로 복귀시키지만, 인프라가 전부 성공한 뒤 **DB 확정 단계에서 실패한 경우에는 PENDING 복귀가 누락**되어 요청이 PROCESSING에 남는 알려진 결함이 있습니다. 실패한 요청이 PROCESSING으로 남아 있으면 인프라 잔재(계정·Pod·NodePort)를 확인한 뒤 상태를 수동 정리해야 합니다.

| 상태 | 뜻 | 인프라 존재 여부 |
|------|-----|----------------|
| PENDING | 신청 접수, 승인 대기 (실패로 되돌린 뒤 복귀한 경우 포함) | 없음 |
| PROCESSING | 승인 진행 중 — 계정·Pod·포트를 만드는 중 | 만들어지는 중 (부분 존재 가능) |
| FULFILLED | 승인 완료, 자원 배정 확정 | Pod·NodePort·계정·홈 존재 |
| DENIED | 관리자 거절 | 없음 (승인 전 거절) |
| DELETED | 삭제 확정 (soft delete) | 없음이 원칙 — 단 외부 정리 실패 시에도 DELETED로 확정되므로 잔재 가능 (운영 가이드 만료 흐름 절 참조) |

만료 시 FULFILLED → DELETED 전이가 실제로 어떤 API 호출을 동반하는지는 [운영 가이드](운영-가이드.md)의 만료 스케줄러 절을 참고합니다.

## 4. 주요 API와 책임 분리

전체 명세는 [API 레퍼런스](API-레퍼런스.md)와 Swagger(`http://210.94.179.18:30082/apidocs/`)에서 확인합니다. 운영 흐름의 핵심 API만 요약합니다.

| API | 역할 |
|-----|------|
| `PUT /accounts/users` | NFS 계정 파일(passwd/group/shadow/sudoers)에 사용자를 추가하고 빈 UID를 골라 배정해 반환합니다 |
| `DELETE /accounts/users/<username>` | passwd·shadow·group(멤버십, 비게 된 그룹 포함)에서 사용자를 제거하고, NAS 홈 삭제(실패해도 계속 진행)와 krb5 정리를 수행합니다. sudoers 정책 파일은 지우지 않습니다 |
| `POST /create-pod` | admin_be 역조회로 사용자 설정을 받아 최적 GPU 노드를 골라 Pod와 NodePort Service를 생성합니다 |
| `POST /delete-pod` | Pod, NodePort Service, NodePort DB row를 정리합니다 |
| `POST /migrate` | 실행 중인 사용자 Pod를 Prometheus 점수가 충분히 더 좋은 노드로 옮깁니다 (사용자 이미지 commit/save 포함) |
| `GET /health` | 헬스체크입니다 (`"OK"`) |

**삭제는 두 API의 합입니다.** `POST /delete-pod`는 Service·NodePort·Pod만 정리하고, 계정 파일·NAS 홈·KDC principal은 `DELETE /accounts/users`의 몫입니다. 하나만 호출하면 반쪽 삭제가 됩니다 — delete-pod만 부르면 계정과 홈이 남고, accounts 삭제만 부르면 주인 없는 Pod가 돌아갑니다. 운영 중 수동 삭제가 필요할 때도 Pod를 kubectl로 직접 지우기보다 이 API를 통해 Service와 DB allocation까지 같이 정리하는 것이 안전합니다. 특히 kubectl로 Pod만 지우면 NodePort Service와 `nodeport_allocations` 배정 행이 그대로 남는데, reconcile(5.1절)은 **Service의 존재**를 기준으로 판단하므로 Service가 살아 있는 한 이 유령 배정을 영영 정리하지 못합니다.

## 5. 포트 할당

### 5.1 포트 배정 구조

NodePort 배정 장부는 infra-mysql `pod_port_db`의 `nodeport_allocations` 테이블입니다. 배정 절차는 동시성 사고를 막는 순서로 짜여 있습니다.

1. 트랜잭션을 열고 `SELECT ... FOR UPDATE`로 사용 중인 포트 행들을 잠급니다. 이 잠금 덕에 동시에 들어온 두 create-pod가 같은 포트를 고르는 일이 없습니다.
2. 30000~32767 범위에서 빈 포트를 골라 insert하고 커밋합니다.
3. 포트당 NodePort Service 1개를 만듭니다. 기본 포트는 22(ssh)와 8888(jupyter)이고, admin_be가 내려준 추가 포트가 있으면 함께 배정합니다.

배정 전에 `reconcile_nodeport_allocations()`가 **MySQL에는 남아 있지만 Kubernetes에는 Service가 없는** — 실제로는 사라졌는데 기록만 남은 — 배정 행을 정리합니다. 장부(MySQL)와 현실(K8s)이 어긋났을 때 현실 쪽을 기준으로 장부를 맞추는 자가 치유 장치입니다. 너무 자주 돌지 않게 5분에 한 번으로 제한되어 있습니다.

두 종류의 포트 개념을 구분해야 합니다.

| 개념 | 저장 위치 | 의미 |
|------|----------|------|
| 요청 포트 (`port_requests`) | admin_be DB | 사용자가 신청서에 적은 컨테이너 내부 포트 (희망 사항) |
| 실제 포트 (`pod_external_ports`) | admin_be DB | create-pod 응답으로 확정된 NodePort 매핑 (사실). FE 접속 정보의 원천입니다 |

"사용자 화면에 뜨는 접속 포트가 이상하다"면 `pod_external_ports`를, "신청서에 적은 포트가 반영이 안 됐다"면 `port_requests` → 역조회 → 배정의 체인을 봅니다. 스키마 상세는 [데이터베이스](데이터베이스.md)를 참고합니다.

### 5.2 방화벽 함정 — 배정 범위와 외부 개방 범위는 다릅니다

클러스터는 30000~32767 범위에서 자유롭게 포트를 골라 배정하지만, **외부(pfSense)에서 실제로 열려 있는 것은 일부뿐**입니다.

- LAB 공인 IP(`210.94.179.18`) — 화이트리스트 8개 포트만 통과합니다.
- FARM 공인 IP(`210.94.179.19`) — farm6 경유 오프셋 매핑입니다. 외부 포트 `9300+(NodePort−30000)`이 NodePort `30000~30097`에 대응합니다. 예를 들어 NodePort 30042는 외부에서 `210.94.179.19:9342`입니다.

즉 배정된 NodePort가 이 범위를 벗어나면 **내부는 정상인데 외부 접속만 안 되는** 상태가 됩니다. 이는 버그가 아니라 정책 공백입니다 — 배정 로직은 방화벽 개방 범위를 모르고, 방화벽은 배정 현황을 모릅니다. "교내에서는 되는데 집에서는 안 돼요"라는 문의가 오면 코드보다 먼저 배정 포트가 개방 범위 안인지 확인합니다(7.2절 그림 참조).

## 6. 계정·스토리지 설계

### 6.1 파일 기반 passwd가 기준 원본인 이유

직관적으로는 계정을 MySQL에 두는 것이 깔끔해 보입니다. 하지만 UID/GID의 존재 이유는 **NFS 파일 소유권**이고, 그 파일들은 FARM 물리 서버(Docker 방식)와 K8s Pod가 공유하는 NAS에 있습니다. 파일 소유권은 디스크에 UID 숫자로 기록되므로([기초 개념](기초-개념.md) 1절), UID 기록을 DB로 분리하면 파일시스템의 실제 소유권과 어긋날 수 있는 두 번째 기록이 생깁니다. 이 어긋남은 곧 UID 충돌 — 남의 홈 소유권 획득 — 이라는 보안 사고입니다.

그래서 "파일 소유권의 기준은 파일시스템 옆에 둔다"를 원칙으로, `/kube_share`의 passwd/group 파일이 기준이 되는 원본(SSOT)입니다. 이 원칙은 코드에도 박혀 있습니다 — admin_be가 계정 생성 요청에 uid를 실어 보내도 config-server는 무시하고 passwd 기준으로 배정합니다.

이 장부가 사용자 컨테이너에 전달되는 방식은 주의가 필요합니다. **현행 Pod 스펙은 `/kube_share` 파일을 컨테이너에 마운트하지 않습니다.** 대신 config-server가 Pod를 만들 때 passwd에서 읽은 값을 환경변수(`TARGET_UID`·`TARGET_GID`·`USER_GROUPS` 등)로 주입하고, 컨테이너 이미지의 entrypoint가 그 값으로 컨테이너 안에서 사용자를 만듭니다. 즉 컨테이너 안의 `/etc/passwd`는 생성 시점에 장부 값으로 찍어낸 사본이며, 장부의 이후 변경이 실시간 반영되지는 않습니다. 계정 파일을 read-only subPath로 직접 마운트하는 방식은 과거 설계로, 코드 주석에 흔적만 남아 있습니다. `nfs_db`(MySQL) + K8s Secret으로 옮기는 개선안은 문서화된 **미구현 계획**이며 현행은 파일 방식입니다.

### 6.2 개별 PVC를 폐기하고 NFS 직접 마운트로 간 이유

초기 설계는 사용자마다 PVC를 만들었으나 26-06-19에 폐기가 결정되고 26-07-02에 운영 반영됐습니다. 폐기 근거는 두 가지입니다.

- **관리 부담과 잔해** — 사용자 수만큼 PVC/PV 객체가 늘어나며 Terminating 잔류, 템플릿 미치환 쓰레기 디렉터리 같은 잔해가 누적됐습니다.
- **격리 착시** — 어차피 모든 PVC가 같은 NFS 서버의 같은 공간을 바라보므로, PVC라는 간접층은 격리를 제공하지 못했습니다. PVC가 격리해 주는 것처럼 보이지만 실질 격리는 처음부터 UID와 `chmod 700`이 담당하고 있었습니다.

그래서 간접층을 제거하고 NFS 공유를 `/home`에 직접 마운트하되, `root_squash` + 홈 사전 생성(NAS SSH)으로 안전을 확보하는 현행 구조가 됐습니다. image-store(사용자 이미지 tar 보관)만 PVC로 남았습니다.

### 6.3 Kerberos와 FARM NFS

FARM NAS 홈 마운트에는 Kerberos(krb5) 인증이 전제됩니다. 이 영역은 별도 담당자 관할이라 이 wiki에서는 다루지 않습니다. 관리자가 알아야 할 사실은 두 가지입니다 — 인증 체인이 끊기면 신규 Pod가 FailedMount로 생성되지 못하며, 이 경우 담당자에게 문의합니다([운영 가이드](운영-가이드.md) 장애 표 참조).

세부를 전부 지우면 이 시스템은 다섯 덩어리입니다.

```mermaid
flowchart LR
    U["사용자·관리자<br/>(외부)"] --> PF["pfSense<br/>(방화벽·공인 IP)"]
    PF --> WEB["정책 계층<br/>admin_fe · admin_be"]
    PF --> POD["사용자 Pod<br/>(ssh · jupyter)"]
    WEB --> CS["실행 계층<br/>config-server + 부속"]
    CS --> POD
    CS --> NAS["NAS<br/>(NFS 저장소)"]
    POD --> NAS
    CS -->|"노드 점수 질의"| MON["모니터링<br/>Prometheus · Grafana"]
```

읽는 법은 이렇습니다. 외부 트래픽은 전부 pfSense를 지나며, 갈래는 두 개뿐입니다 — 웹 화면으로 가는 길과 사용자 Pod(ssh/jupyter)로 가는 길. 정책 계층(admin_fe/admin_be)이 실행 계층(config-server)에 지시하고, 실행 계층이 사용자 Pod를 만들고 지웁니다. 사용자 Pod와 실행 계층은 둘 다 NAS에 얹혀 있고, 모니터링은 실행 계층의 노드 선택에 재료를 공급합니다. 아래 상세 그림들은 이 다섯 덩어리를 하나씩 확대한 것입니다.

### 7.2 상세 — 외부 접근 경로 (pfSense)

```mermaid
flowchart LR
    subgraph EXT["외부 (인터넷)"]
        U1["웹 브라우저<br/>(신청·승인 화면)"]
        U2["ssh / jupyter 클라이언트<br/>(사용자 작업)"]
    end
    subgraph PF["pfSense"]
        LAB["LAB 공인 IP 210.94.179.18<br/>화이트리스트 8개 포트만 통과"]
        FARM["FARM 공인 IP 210.94.179.19<br/>외부 9300+(N-30000) ↔ NodePort N<br/>(N = 30000~30097, farm6 경유)"]
    end
    U1 --> LAB
    U2 --> LAB
    U2 --> FARM
    LAB -->|"30081"| FE["admin_fe"]
    LAB -->|"화이트리스트 내 NodePort만"| UP["사용자 Pod NodePort"]
    FARM -->|"30000~30097 대응분만"| UP
```

이 그림의 핵심은 **공인 IP 두 개의 정책이 다르다**는 것입니다. "내부는 되는데 외부만 안 된다"는 증상은 클러스터가 아니라 이 상자를 의심합니다.

### 7.3 상세 — 정책 계층 (admin_fe / admin_be)

```mermaid
flowchart TB
    FE["admin_fe (정적 서빙)<br/>namespace: ailab-frontend · NodePort 30081"]
    subgraph BE["admin_be (Spring Boot WAS) — namespace: default · Service admin-prod · NodePort 30083"]
        API["신청·승인 API"]
        SCH["만료 스케줄러"]
        MAIL["메일 발송"]
        DB[("admin_be DB<br/>requests · port_requests · pod_external_ports")]
    end
    CS["config-server (ailab-infra)"]
    FE -->|"REST (30083)"| API
    API --> DB
    SCH --> DB
    API -->|"PUT /accounts/users · POST /create-pod"| CS
    SCH -->|"POST /delete-pod · DELETE /accounts/users"| CS
    CS -->|"GET /api/requests/config/... 역조회"| API
    API --> MAIL
```

정책 계층은 장부(requests·port_requests·pod_external_ports)와 스케줄을 쥐고 있습니다. 승인과 만료라는 두 개의 트리거가 각각 config-server API를 호출하고, 반대 방향으로는 역조회 화살표 하나가 돌아옵니다. 역조회가 클러스터 내부 DNS 이름 `admin-prod.default`를 향하므로, admin_be의 Service 이름이나 namespace를 바꾸면 이 화살표가 끊긴다는 점에 주의합니다. 화면·표시 문제는 이 상자(admin_fe/admin_be)에서, 인프라 실패는 다음 상자에서 찾습니다.

### 7.4 상세 — 실행 계층 (namespace: ailab-infra)

```mermaid
flowchart TB
    BE["admin_be"]
    PR["Prometheus (monitoring)"]
    NAS["NAS<br/>SSH 192.168.2.30:6954"]
    subgraph NSI["namespace: ailab-infra"]
        CS["config-server (Flask/gunicorn)<br/>NodePort 30082 ⚠️ 인증 없음"]
        MY[("infra-mysql<br/>pod_port_db.nodeport_allocations")]
        RD[("redis<br/>이미지 작업 상태 img:*")]
        SVC["NodePort Service<br/>(사용자 포트당 1개)"]
        UP["사용자 Pod (ailab-*)"]
    end
    BE -->|"승인·삭제 지시 (30082)"| CS
    CS -->|"역조회"| BE
    CS -->|"SELECT ... FOR UPDATE 포트 배정"| MY
    CS -->|"이미지 commit/save 상태 기록"| RD
    CS -->|"노드 점수 PromQL 질의"| PR
    CS -->|"K8s API로 생성·삭제"| UP
    CS -->|"K8s API로 생성·삭제"| SVC
    SVC -->|"트래픽 전달"| UP
    CS -->|"SSH 홈 사전 생성 + 계정 파일 편집"| NAS
```

config-server를 중심으로 의존선이 다섯 방향으로 뻗습니다 — admin_be(역조회), infra-mysql(포트 장부), redis(이미지 상태), Prometheus(노드 점수), NAS(계정 파일·홈), 그리고 쿠버네티스 API(Pod·Service). **승인 실패의 원인은 거의 항상 이 의존선 중 하나가 끊긴 것**이므로, create-pod 실패 로그를 볼 때 이 그림을 체크리스트로 씁니다. 어느 선이 어떤 증상을 내는지는 [운영 가이드](운영-가이드.md)의 장애 진단 결정 트리와 연결됩니다.

### 7.5 상세 — 사용자 Pod 내부

```mermaid
flowchart LR
    CS["config-server"]
    S1["NodePort Service<br/>(ssh → 22)"]
    S2["NodePort Service<br/>(jupyter → 8888)"]
    S3["NodePort Service<br/>(신청한 추가 포트)"]
    subgraph NODE["GPU 노드 (2.3절 점수로 선택)"]
        subgraph POD["사용자 Pod (ailab-사용자명)"]
            C["컨테이너 — 승인된 이미지<br/>entrypoint가 환경변수 값으로<br/>컨테이너 안 사용자 생성"]
        end
        DEV["GPU 디바이스<br/>/dev/nvidia* (hostPath)"]
        HM["/home — 노드 호스트에<br/>마운트된 NFS (hostPath)"]
    end
    NAS["NAS"]
    CS -->|"생성 시 UID·GID·그룹<br/>환경변수 주입"| C
    S1 --> C
    S2 --> C
    S3 --> C
    C -->|"rw"| HM
    C --- DEV
    HM --- NAS
```

사용자 Pod 하나에 붙는 것은 접속 통로(포트당 Service 1개), 홈 스토리지, GPU 디바이스입니다. `/home`은 **노드 호스트가 마운트해 둔 NFS를 Pod가 hostPath로 물려받는** 구조이고(쓰기 가능), GPU는 승인 개수만큼 `/dev/nvidia*` 디바이스가 hostPath로 붙습니다. 계정 정보는 파일 마운트가 아니라 생성 시 환경변수로 주입됩니다(6.1절). **Pod가 Pending·NotReady에서 멈추면 홈 마운트 문제(FailedMount 계열) 아니면 GPU 자원 부족**이 대부분이므로, `kubectl describe pod`의 이벤트에서 어느 쪽인지 먼저 봅니다.

### 7.6 상세 — 모니터링 (namespace: monitoring)

```mermaid
flowchart LR
    subgraph GPUN["GPU 노드들 (farm*)"]

    end
    subgraph NSM["namespace: monitoring"]
        PR["Prometheus"]
        GF["Grafana<br/>NodePort 30080"]
    end
    CS["config-server"]
    ADM["관리자"]
    PR -->|"주기적 scrape"| DX
    GF --> PR
    CS -->|"노드 점수 PromQL"| PR
    ADM -->|"대시보드 (30080)"| GF
```

모니터링은 조회 전용처럼 보이지만, 이 시스템에서는 **노드 선택이라는 실행 경로에 물려 있습니다.** DCGM exporter가 죽으면 그 노드가 0점(가장 한가함)으로 오인되고, Prometheus 자체가 죽으면 create-pod가 실패합니다(2.3절 폴백). 즉 모니터링 상자의 장애는 "그래프가 안 보임"이 아니라 "승인이 안 됨"으로 나타날 수 있습니다.

### 7.7 상세 — NAS 의존선 (단일 장애점)

```mermaid
flowchart TB
    UP["사용자 Pod"] -->|"/home rw (호스트 NFS 마운트 경유)"| NAS["NAS (NFS 서버)<br/>SSH 192.168.2.30:6954"]
    CS["config-server"] -->|"/kube_share 계정 파일 편집"| NAS
    CS -->|"SSH 홈 사전 생성"| NAS
    MY["infra-mysql"] -.->|"PVC 백엔드 (NFS CSI)"| NAS
    IS["image-store PVC<br/>(사용자 이미지 tar)"] -.-> NAS
```

NAS에는 사용자 홈, `/kube_share` 계정 파일, infra-mysql 데이터, image-store가 전부 걸려 있습니다. 따라서 NAS 장애의 증상은 한 곳에 머물지 않습니다 — 기존 Pod의 홈 I/O가 멈추고, 신규 계정 생성(파일 편집)이 실패하고, 포트 장부(infra-mysql)까지 흔들릴 수 있습니다. **서로 무관해 보이는 여러 상자가 동시에 이상하면 NAS부터 봅니다.**

### 7.8 장애 시 이 지도를 읽는 법

- 내부(교내망)는 정상인데 외부 접속만 안 되면 → 7.2 (pfSense 개방 범위).
- 화면·표시·상태 값 문제 → 7.3 (admin_fe / admin_be와 그 장부).
- 승인·Pod 생성 실패 → 7.4의 의존선을 체크리스트로 (역조회·MySQL·Prometheus·NAS·K8s API).
- Pod가 Pending/FailedMount → 7.5 (마운트 두 가닥, GPU 자원).
- 특정 노드로 Pod 쏠림, 혹은 create-pod가 노드를 못 고름 → 7.6 (exporter·Prometheus).
- 여러 증상 동시다발 → 7.7 (NAS 단일 장애점).
